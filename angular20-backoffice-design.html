Angular 20.x Released May 28, 2025 Updated from v17 guide

Complete implementation design for the mechanic shop manager dashboard — rewritten for Angular 20, with signal-based inputs/outputs, httpResource() for data fetching, stable effect() + linkedSignal(), and zoneless change detection.

What changed from Angular 17 → 20

Angular 20 is a focused polish release. No architecture revolution — but several APIs that were experimental are now stable, and one new API (httpResource()) fundamentally changes how you fetch data.

The big one: httpResource()

httpResource() is a reactive wrapper around HttpClient that exposes request state — value, loading, error, headers — all as signals. It replaces the service.getAll().subscribe(data => signal.set(data)) pattern with a single declarative line. It's experimental in v20 but production-safe with predictable semantics.

RxJS role in Angular 20

RxJS is no longer central to data fetching — httpResource() handles that reactively with signals. Keep RxJS only for mutations (POST/PUT/DELETE via HttpClient) and complex multi-stream operations. GET calls should use httpResource().

Services now: mutations only

In Angular 17, services returned Observables for GET requests which were then subscribed to in the store. In Angular 20, GET calls move into httpResource() inside the store. Services now only handle mutations (POST, PATCH, DELETE) using HttpClient directly. This is simpler and more reactive.

httpResource() — reactive data fetching new in v20

httpResource() replaces the service.getAll().subscribe() pattern. It wraps HttpClient and exposes the response, loading state, and errors as signals you can use directly in templates and computed signals.

Core API

// How httpResource works
import { httpResource } from '@angular/common/http';

// Static URL
workOrders = httpResource<WorkOrder[]>('/api/work-orders');

// Reactive URL — re-fetches when the signal changes
selectedId = signal('WO-441');
detail = httpResource<WorkOrder>(
  () => `/api/work-orders/${this.selectedId()}`
);

// Reading the resource signals in a template or computed
const data      = this.workOrders.value();       // WorkOrder[] | undefined
const isLoading = this.workOrders.isLoading();  // boolean
const error     = this.workOrders.error();      // unknown | undefined
const hasValue  = this.workOrders.hasValue();   // boolean

httpResource API
httpResource is for GET only

Use httpResource() for GET (read) requests. For mutations — creating work orders, updating status — continue using HttpClient directly in a service, returning an Observable. Call .reload() on the resource after a mutation to re-fetch fresh data.

DashboardStore rewritten for v20

The store is completely rewritten. Instead of injecting services and calling forkJoin().subscribe(), it declares httpResource() instances and exposes computed signals. The store becomes declarative rather than imperative.

Angular 17 — imperative 
export class DashboardStore {
  woSvc = inject(WorkOrderService);

  workOrders = signal<WO[]>([]);

  loadAll() {
    forkJoin({
      workOrders: this.woSvc.getAll()
    }).subscribe(d =>
      this.workOrders.set(d.workOrders)
    );
  }
}
Angular 20 — declarative 
export class DashboardStore {
  // Declare resource — it fetches on init
  private woRes =
    httpResource<WO[]>('/api/work-orders');

  // Expose with fallback
  workOrders = computed(() =>
    this.woRes.value() ?? []
  );
  // No loadAll() needed!
}

Full DashboardStore

// core/store/dashboard.store.ts
import { Injectable, computed, inject } from '@angular/core';
import { httpResource } from '@angular/common/http';
import type { WorkOrder, Customer, InventoryItem, Technician } from '@core/models';

@Injectable({ providedIn: 'root' })
export class DashboardStore {

  // ── HTTP Resources (fetched automatically on init) ──────────
  private readonly woResource   = httpResource<WorkOrder[]>('/api/work-orders');
  private readonly custResource = httpResource<Customer[]>('/api/customers');
  private readonly invResource  = httpResource<InventoryItem[]>('/api/inventory');
  private readonly techResource = httpResource<Technician[]>('/api/technicians');

  // ── Public data signals (with empty fallbacks) ──────────────
  workOrders  = computed(() => this.woResource.value() ?? []);
  customers   = computed(() => this.custResource.value() ?? []);
  inventory   = computed(() => this.invResource.value() ?? []);
  technicians = computed(() => this.techResource.value() ?? []);

  // ── Global loading state ─────────────────────────────────────
  loading = computed(() =>
    this.woResource.isLoading() ||
    this.custResource.isLoading() ||
    this.invResource.isLoading()
  );

  // ── Computed aggregates ─────────────────────────────────────
  urgentCount = computed(() =>
    this.workOrders().filter(wo => wo.status === 'urgent').length
  );

  lowStockItems = computed(() =>
    this.inventory().filter(i => i.quantity < i.minQuantity)
  );

  monthlyRevenue = computed(() =>
    this.workOrders().reduce((acc, wo) => acc + wo.value, 0)
  );

  avgTechUtilization = computed(() => {
    const t = this.technicians();
    return t.length
      ? t.reduce((a, t) => a + t.utilization, 0) / t.length
      : 0;
  });

  // ── Reload (call after mutations) ────────────────────────────
  reloadWorkOrders() { this.woResource.reload(); }
  reloadInventory()   { this.invResource.reload(); }
}

Mutation services simplified in v20

Services no longer need to expose getAll() — that moved to httpResource() in the store. Services now only handle write operations and call store.reload*() when done.

// core/services/work-order.service.ts
@Injectable({ providedIn: 'root' })
export class WorkOrderService {
  private http  = inject(HttpClient);
  private store = inject(DashboardStore);
  private readonly base = '/api/work-orders';

  // Mutations only — GET is handled by httpResource in the store

  create(wo: Partial<WorkOrder>): Observable<WorkOrder> {
    return this.http.post<WorkOrder>(this.base, wo).pipe(
      tap(() => this.store.reloadWorkOrders())   // refresh store
    );
  }

  updateStatus(id: string, status: WorkOrderStatus): Observable<WorkOrder> {
    return this.http.patch<WorkOrder>(`${this.base}/${id}`, { status }).pipe(
      tap(() => this.store.reloadWorkOrders())
    );
  }
}

signal-based input() / output() stable

All shared components use input() and output() instead of @Input() and @Output() decorators. The values returned by input() are signals — you call them with () to read them.

Angular 17 — decorators 
export class KpiCardComponent {
  @Input({required:true})
  label!: string;

  @Input()
  trend: 'up'|'down'|'neutral'
    = 'neutral';

  @Output()
  clicked = new EventEmitter<void>();
}
Angular 20 — signals 
export class KpiCardComponent {
  label = input.required<string>();

  trend = input<'up'|'down'|'neutral'>(
    'neutral'
  );

  clicked = output<void>();
}
Reading inputs in the class vs template

Input signals are read by calling them: this.label() in the class, or {{ label() }} in the template. In computed signals they participate in dependency tracking automatically: computed(() => this.label().toUpperCase()) re-runs whenever the parent passes a new value.

DataTable component

export interface ColumnDef<T> {
  key:       keyof T;
  header:    string;
  render?:   (val: T[keyof T], row: T) => string;
  sortable?: boolean;
}

@Component({
  selector: 'app-data-table',
  standalone: true,
  template: `
    <table class="tbl">
      <thead><tr>
        @for (col of columns(); track col.key) {
          <th (click)="col.sortable && sort(col.key)"
              [class.sortable]="col.sortable">
            {{ col.header }}
            @if (sortKey() === col.key) {
              {{ sortAsc() ? '↑' : '↓' }}
            }
          </th>
        }
      </tr></thead>
      <tbody>
        @for (row of sortedRows(); track $index) {
          <tr (click)="rowClick.emit(row)">
            @for (col of columns(); track col.key) {
              @let val = row[col.key];
              <td>{{ col.render ? col.render(val, row) : val }}</td>
            }
          </tr>
        }
      </tbody>
    </table>
  `
})
export class DataTableComponent<T> {
  columns = input.required<ColumnDef<T>[]>();
  rows    = input.required<T[]>();
  rowClick = output<T>();

  // Local sort state — private signals
  sortKey = signal<keyof T | null>(null);
  sortAsc = signal(true);

  sortedRows = computed(() => {
    const k = this.sortKey();
    if (!k) return this.rows();
    return [...this.rows()].sort((a,b) =>
      (a[k] > b[k] ? 1 : -1) * (this.sortAsc() ? 1 : -1)
    );
  });

  sort(key: keyof T) {
    if (this.sortKey() === key) this.sortAsc.update(v => !v);
    else { this.sortKey.set(key); this.sortAsc.set(true); }
  }
}

Work Orders page — linkedSignal for filter

linkedSignal() is now stable in Angular 20. It creates a writable signal whose default value is derived from another signal, but which can also be mutated locally. It's perfect for filter state that should reset if the underlying data changes.

@Component({ selector: 'app-work-orders', standalone: true, /* ... */ })
export class WorkOrdersComponent {
  store = inject(DashboardStore);

  // linkedSignal — resets to 'all' when workOrders data changes
  activeFilter = linkedSignal<WorkOrder[], WorkOrderStatus | 'all'>({
    source: this.store.workOrders,           // trigger: re-run on new data
    computation: () => 'all' as const        // always reset to 'all'
  });

  filtered = computed(() => {
    const f = this.activeFilter();
    const all = this.store.workOrders();
    return f === 'all' ? all : all.filter(wo => wo.status === f);
  });

  countByStatus = (s: WorkOrderStatus) =>
    computed(() => this.store.workOrders().filter(w => w.status === s).length);

  setFilter(f: WorkOrderStatus | 'all') {
    this.activeFilter.set(f);   // linkedSignal is writable
  }
}

Charts (ng2-charts)

No change from the v17 approach. Each chart is its own standalone component. Use effect() (now stable) to update chart data reactively when store signals change.

@Component({ selector: 'app-revenue-chart', standalone: true, imports: [BaseChartDirective],
  template: `<canvas baseChart [data]="chartData" [options]="opts" [type]="'bar'"></canvas>`
})
export class RevenueChartComponent {
  private store = inject(DashboardStore);

  chartData = signal<ChartData<'bar'>>({
    labels: ['Jan','Feb','Mar','Apr','May','Jun'],
    datasets: [{ label: 'Revenue', data: [] }]
  });

  // effect() is now STABLE — use it without reservations
  private _ = effect(() => {
    const wo = this.store.workOrders();
    this.chartData.update(d => ({
      ...d,
      datasets: [{ label: 'Revenue', data: groupByMonth(wo) }]
    }));
  });

  opts: ChartOptions<'bar'> = {
    responsive: true, maintainAspectRatio: false,
    plugins: { legend: { display: false } },
    scales: {
      x: { grid: {display:false}, border:{display:false} },
      y: { ticks: { callback: v => '$' + (v as number/1000).toFixed(0) + 'k' } }
    }
  };
}

In angular.json under architect.build.options.polyfills, remove "zone.js" once you switch to provideZonelessChangeDetection(). This reduces your bundle by ~30KB and eliminates Zone.js patching of browser APIs.

Run ng new, set up tsconfig path aliases, add _tokens.scss, configure app.config.ts with zoneless + httpClient + charts providers.

2

Models

Write all interfaces. Create MOCK_* constants for development. This is the shared contract everything depends on.

3

DashboardStore with mock httpResource

Implement the store with httpResource(). During development, Angular's provideHttpClient(withInterceptors([mockInterceptor])) pattern lets you intercept and return mock data without changing the store code.

4

Shell + routing

Build shell, sidebar, topbar. Set up all routes. Confirm navigation works with empty feature stubs.

5

Shared components with input()

Build KpiCardComponent, BadgeComponent, SparkBarComponent, AlertItemComponent, DataTableComponent. All use input() / output(). Test in isolation.

6

Overview page first

Build the full Overview layout — uses all shared components and validates the architecture end to end before you build the other pages.

7

Chart components with effect()

Build each chart component. Use effect() (now stable) to reactively update chartData signal when store data changes.

8

Remaining feature pages

Work Orders (with linkedSignal filter) → Customers → Inventory → Staff. Each follows the same inject-store, compute-local, compose-shared pattern.

9

Mutation services

Add POST/PATCH/DELETE methods to services. Wire the "New work order" button. After each mutation, call store.reloadWorkOrders() to refresh via httpResource.

10

Swap mock interceptor for real API

Remove the mock HTTP interceptor. Point httpResource() URLs at the real backend. Nothing else changes — that's the payoff of the resource pattern.

11

Enable zoneless

Switch to provideZonelessChangeDetection(), remove zone.js from polyfills, run the app, fix any change detection issues (components that mutated state outside signals — rare if you built signal-first from the start).

Core Angular 20 principle

Declare, don't subscribe. In Angular 20, the ideal component has no ngOnInit, no subscribe(), and no manual .set() calls for data loading. Data comes in through httpResource(), flows through computed(), and renders through signal reads in templates. Side effects use effect() only when truly necessary.
